# Demo 全交互真实化 — 现状审计与研发计划

> 日期：2026-07-20｜触发：王老师要求核实"所有按钮、所有交互是否都有实际功能"，如未完成需给出研发计划。
> 结论先行：**未完成**。当前系统是"视觉完成、交互仿真"的汇报演示版（PRD 状态 F001-F009 标 🟢 是相对"演示验收标准"而言成立的），但按王老师现在提出的"每个按钮都要有实际功能"这一更高标准衡量，还有明显缺口。

## 一、审计方法

逐页读取全部 9 个页面源码（`src/app/(platform)/**/page.tsx`，共 851 行）与共享组件，对每一个可交互元素（Button / Select / Input / DatePicker / Segmented）做三分类：

| 分类 | 定义 |
|------|------|
| ✅ 真实交互 | 点击后有可观察的状态变更（本页 `useState` 真实改变） |
| 🟡 仅提示无状态 | 点击只弹 `message.success` 固定文案，页面数据/UI 不变 |
| ⚪ 完全无绑定 | 没有 `onClick`/`onChange`，纯装饰控件 |

同时确认：**没有任何 API 路由**（`src/app/api` 不存在）、**没有 `.env`/数据库连接**（Prisma 仅有 schema，未接入），所有数据来自 `src/lib/demo-data.ts` 静态数组，且**每个页面各自 import 一份**，互不联动。

## 三、比"死按钮"更本质的问题：没有联动

即使是审计里标为"✅ 真实交互"的 6 处，也只改了**当前页面自己的 `useState`**，不改共享数据源，刷新即丢失，且不会传导到其他相关页面。例如：

- `/quality` 点"复验通过"关闭一个质量问题 → `/dashboard` 的"未关闭质量问题 7 项"KPI 不变，`/vehicles/[id]` 证据包里同一条问题也不会变成"已关闭"。
- `/review` 点"冻结 V4.0" → `/projects/[id]`"当前执行依据：改制方案 V3.0"、`/vehicles/[id]`"执行方案 V3.0 已冻结"标签都不会变成 V4.0。
- `/workshop` 扫码/报工推进的进度 → `/dashboard`"核心项目完成率 68%"、`/projects/[id]` 进度条不会同步。

这正是 CLAUDE.md「Web 系统规则」第 2 条要求的"生命周期类功能必须同时检查列表、详情、操作入口、看板和一车一档是否同步"——目前**没有一处真正做到跨页同步**，因为九个页面各自 import 静态数组，不存在共享的可写数据源。

## 四、研发计划

### Phase 0：共享数据层（前提，约 0.5 天）

不引入新依赖（当前未装状态管理库，`karpathy-guidelines` 六级阶梯第 4 档：已装依赖能解决就不新增）：

- 用 React `Context + useReducer` 把 `src/lib/demo-data.ts` 的数组包成一个 `DemoDataProvider`，在 `app-providers.tsx` 里全局挂载。
- reducer 覆盖当前所有需要变更的动作：`FREEZE_VERSION`、`CLOSE_REVIEW_COMMENT`、`VERIFY_QUALITY_ISSUE`、`CREATE_QUALITY_ISSUE`、`ADJUST_SCHEDULE_SLOT`、`SCAN_PART`、`REPORT_PROGRESS`、`RECEIVE_MATERIAL`、`RETRY_INTEGRATION`、`CREATE_PROJECT_TASK` 等。
- 加 `localStorage` 同步（写时 `setItem`，Provider 挂载时 `getItem` 回填），保证刷新不丢失；页头提供一个"重置演示数据"入口，方便汇报前恢复初始状态。
- 各页面把 `import { xxx } from "@/lib/demo-data"` 改为 `useDemoData()` 读取同一份数据。

### Phase 1：把全部按钮改造为真实交互 + 跨页联动（约 3-4 天，按页拆分，可并行）

逐页把"⚪ 完全无绑定"和"🟡 仅提示无状态"两类改为调用 Phase 0 的 reducer 动作，并确认下游页面同步：

1. **dashboard**：导出汇报快照 → 前端生成当前 KPI/风险快照（打印视图或 CSV/JSON 下载），不依赖后端也应有真实产物。
2. **projects**：新建任务接 Modal + Form，提交后 `dispatch(CREATE_PROJECT_TASK)` 追加到列表；搜索框/类型下拉/状态下拉接真实前端过滤（`Array.filter`，无需后端）。
3. **projects/[id]**：生成项目快照同 dashboard 导出逻辑，取该项目切片数据。
4. **review**：冻结动作改为 `dispatch(FREEZE_VERSION)`，联动 `projects/[id]`、`vehicles/[id]` 的"执行方案"展示；"查看 V3.0 执行版"打开只读预览 Modal（可先用占位文档链接）。
5. **schedule**：重新计算/采纳建议/调整该班次改为真实修改 `scheduleRows` 对应槽位并重算冲突提示消失逻辑；DatePicker/日周月切换至少做到真实过滤展示的日期文案（不要求真实多日数据也需先声明范围）。
6. **materials**：打印件码/车间接收改状态位（如"待接收"→"已接收"）；查看箱内清单改为真实 Drawer/Modal 展示清单内容而非仅 toast。
7. **workshop**：暂停任务真实切换任务状态（并联动 dashboard/看板）；质量点检按钮改为真实递增点检计数（而不是写死"9/12"文案）；查看冻结文件包打开真实文件列表 Modal；PDA 模式切换至少做到真实切换展示布局或提示当前模式。
8. **quality**：新建质量问题接 Modal+Form 真实入库；导出问题清单做真实 CSV 导出；分类下拉接真实前端过滤；"复验通过"的关闭结果需要联动 dashboard 未关闭计数与 vehicles/[id] 证据包同一条目状态；非复验行"查看"打开详情 Modal。
9. **integrations**：健康检查改为真实刷新所有系统卡片状态（可用模拟延迟+随机/固定结果，但要有真实前后差异）；重新处理把该条日志状态真实改为 healthy；连接器配置至少打开一个真实的配置态 Modal（列出连接器参数，即使是只读）。
10. **vehicles/[id]**：车辆二维码用前端二维码库（如已装依赖没有则用 canvas 手写生成，避免为一个按钮引入重依赖）生成真实可扫码图片；导出车辆档案做真实 PDF/打印视图导出，聚合该车辆当前（联动后）的最新状态。

每页改完后按 CLAUDE.md 规则逐一验证：改动是否在 dashboard 看板、项目详情、一车一档三处同步可见。

### Phase 2（可选，P1，需王老师先拍板再排期）：真正后端持久化

对应 PRD 里已经标注 🔴 的 F010（认证权限）、F011（真实数据库）：

- 引入 Next.js API Routes + Prisma + MySQL（复用 `prisma/schema.prisma` 现有模型），把 Phase 0/1 的 Context+localStorage 换成真实接口调用，实现刷新、多人协作、跨浏览器数据一致。
- 需要先确认：是否要在当前 Next.js 栈上做，还是等客户对招标书要求的 Vue3+SpringBoot 栈给出最终答复（`docs/PRD.md` 五、技术决策已列为待确认项）——避免这一步做完后因技术栈变化被推倒重做。
- 补登录与角色权限（NextAuth 或轻量 session），区分 PRD 里列出的 7 类角色可见范围。

**建议**：Phase 0+1 不依赖后端决策、成本可控（一周内），先做；Phase 2 涉及技术栈拍板和更大投入，建议单独跟王老师确认时机后再排期，不纳入本轮。
