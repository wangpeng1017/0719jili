# 吉利试制样车改制数字化平台深度需求分析

> 分析日期：2026-07-19｜用途：客户汇报、产品范围确认和后续正式 PRD 基线

## 1. 分析材料与可信度排序

本次分析覆盖项目目录内的三层材料：

1. `交付文件/吉利试制样车改制数字化项目招标技术要求书_V15.docx`：494 行提取文本，含 FR-001 至 FR-1008、集成要求和技术要求。作为最新、结构化程度最高的需求基线。
2. `交付文件/吉利试制样车改制数字化解决方案 - V13.pptx`：45 页、1777 行提取文本，含业务理解、角色、模块、实施路线、案例和会议更新。作为解决方案口径和汇报叙事基线。
3. `sources/改制数字化需求 (1).md`：原始业务调研材料，含线下流程、角色访谈、痛点、输入输出物和大量待确认项。作为需求来源证据，用于识别最新文档中被简化或改写的业务细节。

可信度原则：最新招标书 > 最新方案 > 原始调研；但发生范围冲突时不能静默选择，必须列入客户确认清单。

## 2. 项目本质判断

这不是普通的“项目管理系统”，也不是完整替代 TOCC、SAP、LES 的大而全平台。它的准确定位是：

> 以车辆 UID 为主线、以改制任务为驱动，衔接上游车辆/项目总账与下游车间执行证据链的试制改制轻量 MES。

系统的核心价值不在多建几个台账，而在解决五类断点：

- 方案断点：PPT、Excel、群消息中的版本与评审意见无法形成执行依据。
- 计划断点：项目计划与举升机、工位、班组、物料等资源计划相互分离。
- 实物断点：新装件、拆车件、箱/托盘与目标车辆之间缺少可追溯绑定。
- 质量断点：质量策划介入晚，问题、整改、复验、横展与放行没有系统门禁。
- 履历断点：交付后外地换件和返修无法继续补充一车一档。

## 3. 必须跑通的业务主链路

```mermaid
flowchart LR
  A[TOCC需求/WBS/车辆] --> B[任务受理与分类]
  B --> C[车辆范围和身份校验]
  C --> D[结构化方案编制]
  D --> E[页级评审与版本冻结]
  E --> F[六项生产准备门禁]
  F --> G[项目+资源双线排产]
  G --> H[工位扫码/点检/报工]
  H --> I[质量问题整改复验]
  I --> J[终检与交付包]
  J --> K[一车一档与TOCC回写]
  K --> L[试验换件/返修回流]
```

下周汇报不应平均讲所有模块，建议围绕一台主车辆 `GEELY-VH-7E001` 演示完整链路；每个页面都回答“这一步的门禁是什么、证据在哪里、失败后怎么处理”。

## 4. 九个核心业务域

| 业务域 | 要解决的问题 | 关键业务对象 | 核心门禁/规则 |
|--------|--------------|--------------|---------------|
| 需求与任务 | 多来源需求重复录入、责任不清 | WBS、项目、改制任务、车辆范围 | 来源版本留痕；重复车辆任务提示 |
| 车辆档案 | 身份复杂、多次改制履历分散 | 车辆 UID、VIN、样车号、TOCC ID | 至少一个权威身份命中；未完结任务冲突校验 |
| 方案评审 | PPT 线下流转、意见和版本失控 | 方案版本、章节、页面、批注、附件 | 未关闭意见不可冻结；冻结版不可覆盖 |
| 生产准备 | 开工条件靠口头确认 | 车辆、方案、文件、物料、资源、质量策划 | 六项门禁任一失败不可开工 |
| 计划排产 | 项目窗口与稀缺资源脱节 | 计划窗口、班次、工位、举升机、班组 | 插单必须计算影响并记录人工调整原因 |
| 现场执行 | 纸质文件、扫码与报工证据不足 | 车辆任务、件码、工位、点检、报工 | 车辆+任务+工位+件码一致；未点检不可过点 |
| 物料追溯 | 四类物料状态不统一 | 标准件、调拨件、新开件、拆车件、箱码 | 未接收不可装车；报废件锁定；回装校验来源车 |
| 质量闭环 | 问题只记录描述、不闭环 | 问题、责任、根因、措施、复验、横展、让步 | 未复验不可关闭；未关闭/未让步不可交付 |
| 交付与回流 | 交付资料靠人工拼接、交付后断链 | 交付包、一车一档、换件事件、回写日志 | 交付包完整、问题关闭、版本冻结后才可交付 |

## 5. 角色与工作台设计

最新方案形成六类核心角色，页面入口应按职责而不是按数据库表组织：

| 角色 | 首页最重要的信息 | 高频动作 |
|------|------------------|----------|
| 试制策划经理 | 项目进度、方案冻结、交付风险 | 受理需求、组织评审、跟踪版本 |
| 工艺工程师 | 待编制文件、评审意见、现场异常 | 编制工艺卡、回复意见、发布冻结文件 |
| 生产平衡工程师 | 举升机/工位占用、冲突、缺料 | 排产、插单、调整班组和资源 |
| 质量工程师 | 待检任务、未关闭问题、横展 | 策划检查项、分派问题、复验与放行 |
| 改制技师 | 当前车辆任务、冻结文件、扫码点检 | 开工、拆装件扫码、异常反馈、报工 |
| 物料管理专员 | 齐套、配送、接收、拆车件容器 | 接收、配送、绑定箱码、维护库位 |

管理驾驶舱是跨角色的汇总入口，不应取代上述角色工作台。

## 6. 关键数据模型

系统需要围绕关联关系设计，而不是把现有 Excel 原样搬上网：

- `RetrofitProject`：WBS/项目级承诺与组合视图。
- `Vehicle`：车辆唯一身份和一车一档聚合根。
- `RetrofitTask`：车辆级、阶段级执行单元。
- `SolutionVersion`：结构化方案、附件、页级评审和冻结状态。
- `ScheduleSlot`：车间、资源、班次和任务占用。
- `MaterialRequirement`：车辆/任务级物料需求与齐套状态。
- `PartTraceEvent`：拆件、装件、入箱、移库、回装、报废事件。
- `QualityIssue`：问题分类、责任、根因、措施、复验和横展。
- `DeliveryPackage`：实车、文件、物料证据、质量证据和回写状态。
- `IntegrationLog`：外部系统、业务键、数据版本、结果、异常与重试。

## 7. 最重要的业务不变量

这些规则应成为后续接口和数据库设计的自动校验，而不是培训口号：

1. 所有现场动作必须引用明确车辆、任务、工位、操作人和冻结方案版本。
2. 方案评审与计划确认相互关联但不能合并；评审通过不等于交期承诺。
3. 冻结版本只读；任何执行依据变更必须升版并重新评估物料、排产、在制任务和质量点。
4. 标准项目可关联多车，但每台车必须独立排产、执行、质量和交付。
5. 现场流程保持柔性：按任务类型配置阶段和检查项，不强制所有业务使用固定流水工序。
6. 排程推荐必须保留人工干预；人工调整必须记录前后差异和原因。
7. 物料不齐既要提示项目风险，也要按车辆/阶段判断是否真正阻断当前开工。
8. 质量问题必须完成整改与复验；需横展的问题未完成横展时不得批量交付。
9. 一车一档不是报表，而是需求、版本、计划、执行、实物和质量事件的可回放时间线。

## 8. 需求材料中的范围冲突

### 8.1 车辆与项目管理边界

原始调研明确“项目管理/车辆信息管理承接在 TOCC 二期，不包含在改制产品”；V15 又要求车辆建档、一车一档、项目总览。

建议解释为：TOCC 保留权威主档，本系统必须保存业务所需的车辆/项目投影和改制履历，不重新成为主数据责任方。

### 8.2 改制 BOM 边界

原始调研因规则未固化，将上游改制 BOM 管理排除；V15 要求冻结方案形成任务/车辆物料需求，并与 SAP/LES 协同。

建议一期只管理“改制执行 BOM/物料需求快照及版本”，不承诺替代 GBOM/SAP 的主数据和完整工程 BOM 管理。

### 8.3 LIMS 与 TOCC 的需求入口

早期材料大量描述 LIMS 委托单转任务单；最新口径以 TOCC 需求、WBS、车辆为主来源，同时保留 LIMS/Excel 过渡入口。

建议汇报时表述为“TOCC 主入口，LIMS/Excel 可配置过渡入口”，避免把接口关系讲死。

### 8.4 技术栈冲突

招标技术要求书指定 Vue 3 + Vite 3 + Pinia + Element Plus、Spring Boot 3.5 + Spring Security；本项目参考 LIMS-Next，采用 Next.js 15 + React 18 + TypeScript + Ant Design。

**已确认（2026-07-21）：招标指定技术栈为可协商条款，允许等价技术栈。** 因此当前 Next.js 全栈架构可作为正式交付基础继续演进，无需按指定栈重建。后续仅需在合同/技术协议中书面固化“等价技术栈”这一约定。

## 9. 下周汇报版范围取舍

### 已实现 P0/P1

- 管理驾驶舱
- 项目与车辆级任务
- 方案 V4.0 页级评审、意见关闭和冻结演示
- 六项开工门禁
- 三区域举升机 AM/PM/EV 调度板
- 四类物料齐套和拆车件容器追溯
- 数字工位扫码、点检和报工反馈
- 质量问题复验关闭
- 一车一档时间线与证据包
- TOCC/SAP/LES 集成状态和日志

### 有意不在本周实现

- 真正的在线 PPT 编辑器、坐标标注和版本差异算法
- AI 生成方案、2D/3D 图纸比对
- 自动排程求解器和拖拽持久化
- SSO、NextAuth、角色/数据权限后台
- MySQL 数据持久化和真实外部接口
- 动态表单、流程设计器和可视化看板设计器

这些能力工作量大且不会显著提高下周第一次汇报的说服力，应在客户确认试点范围后分阶段建设。

## 10. 正式实施前必须确认的问题

1. 招标指定技术栈是否为强制验收项。
2. TOCC 二期可提供的车辆、项目、WBS、状态字段与接口时间表。
3. LIMS 委托单是否仍是一期正式入口，还是仅作为过渡数据源。
4. 改制 BOM 的责任系统、版本主责和 SAP 交互方式。
5. LES 是否覆盖线边库；若不覆盖，线边库存由本系统新建到什么颗粒度。
6. 拆车件中哪些必须单件追溯，哪些允许箱/托盘级追溯。
7. 三个车间的实际举升机、工位、班次和跨车间调度规则。
8. 六项门禁的强阻断与软提醒清单，以及让步审批人。
9. 质量问题分类、横展规则、终检模板和客户验收签字方式。
10. SSO、组织权限、文件存储、日志保留、附件大小、并发量和弱网要求。
11. 10 月底上线目标对应的试点车型、车辆数量和验收样例。

## 11. 建议的客户汇报顺序

1. 驾驶舱：先讲管理层今天看不清什么。
2. 重点项目：展示车辆级进度和六项门禁。
3. 方案评审：演示意见关闭后 V4.0 才能冻结。
4. 排产：展示举升机冲突与系统建议，强调人工可干预。
5. 物料与工位：展示缺料风险、拆车件容器和扫码证据。
6. 质量：演示复验关闭如何解除交付门禁。
7. 一车一档：回到同一台车，证明数据链路贯通。
8. 集成中心：最后说明 TOCC/SAP/LES 各自责任，避免客户认为重复建设。

建议把“系统能否把一台车从需求讲到交付”作为汇报成功标准，而不是页面数量。
