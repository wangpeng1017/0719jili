# 吉利试制样车改制数字化平台 差距分析（GAP Analysis）

> 编制日期：2026-07-21｜用途：从"业务主线演示版"演进为"试制改制轻量 MES"的迭代基线
> 汇报节奏：下月汇报，本月做真实迭代（非演示冲刺）

## 0. 结论先行

当前版本是**业务主线演示版（Demo）**：它把"需求承接 → 方案评审 → 冻结 → 排产 → 物料齐套 → 质量闭环 → 一车一档"这条**管理链路**讲通了，但客户每天真正要用的**车间执行层（工艺 / 工序 / 生产执行 / 报工 / SOP）基本是模拟的，不是真功能**。

客户原始调研报告的核心判断是：

> "缺乏 MES 系统支持，生产计划、车辆跟踪均靠人工处理。"

客户要的不是更多看板，而是一套能支撑车间真实作业的轻量 MES。本文件逐域对照客户原始需求与当前实现，给出覆盖度评级与演进路线。

## 4. 用户点名的五大缺口（深度说明）

### 4.1 工艺（❌ 完全缺失）
当前只有"方案附件"，没有**工艺路线 / 工艺卡**这一业务对象。客户要求工艺工程师线上编制作业指导书、版本可追溯、设变前版本可查。需新建工艺模型：工艺路线（阶段→工序→检查项→扫码节点）+ 工艺卡（作业步骤、扭矩/参数项、附件）+ 版本冻结。

### 4.2 工序（🟡 仅骨架）
系统只有**固定**七阶段（接车/拆/改/装/调/检/交付），不可按任务类型配置，无工序级路线、无**转序**（FR-503）。客户明确"不强制固定流水工序，按任务类型配置阶段/子任务/检查项/扫码节点"。需把阶段改为可配置工序模板。

### 4.3 生产执行（⚠️ 模拟，非真实）
workshop 页是演示模拟：扫码 scanCount+1、报工 progress+8、点检 counter+1。没有真实状态机（下达→派工→开工→暂停→完工→转序）、没有资源占用/释放（FR-504）、没有在制时长实时 tracking（FR-505）。FR-501~510 大部分未真实实现。

### 4.4 报工（⚠️ 严重简化）
"报工"只是进度条 +8，没有：节点扫码（上线/阶段开始/完成/转序/下线）、测量值/扭矩记录、操作人绑定、时间证据。客户要的是结构化报工数据（关键扭矩、三坐标、操作人、时间），作为一车一档的执行证据。

### 4.5 SOP / 作业指导书（🟡 仅展示）
工位页能"看"冻结方案，但没有结构化 SOP：无作业步骤、无扭矩/参数项、无按步骤确认。客户要求技师在工位查询冻结工艺/质量文件并**按其作业**。需把工艺卡/质量文件结构化为可逐步确认的作业指导。

## 6. 演进路线（下月汇报，本月真实迭代）

按客户痛点价值排序，分三步：

### 第一步：工艺 / 工序 / SOP 底座（客户最痛：工艺文件线下打印）
- 工艺路线模型：阶段 → 工序 → 检查项 → 扫码节点，按任务类型配置
- 结构化 SOP / 工艺卡：作业步骤 + 扭矩/参数项 + 附件，版本冻结
- 质量文件模板编制
- 工位端按冻结 SOP 逐步作业

### 第二步：真实生产执行 + 报工
- 任务状态机：下达 → 派工 → 开工 → 暂停 → 完工 → 转序
- 资源占用/释放（工位/举升机），在制时长实时看板
- 结构化报工：节点扫码 + 测量值/扭矩 + 操作人 + 时间证据
- 过点信息、过程异常实时展示

### 第三步：质量文件 / 横展 + 平台能力
- 结构化检验（按质量文件模板），横展清单联动
- 动态表单 + 可视化流程引擎（满足 V15 6.1）
- 消息中心 / 文件中心
- 集成连接器配置（TOCC/SAP/LES 真实对接准备）

## 7. 演进前仍需客户确认的事项

1. 三个车间的实际举升机、工位、班次和跨车间调度规则
2. 六项开工门禁的强阻断 vs 软提醒清单，让步审批人
3. 拆车件中哪些必须单件追溯、哪些允许箱/托盘级追溯
4. 质量问题分类、横展规则、终检模板、客户验收签字方式
5. TOCC 二期可提供的车辆/项目/WBS 字段与接口时间表
6. 改制 BOM 责任系统、版本主责与 SAP 交互方式
7. 10 月底上线目标对应的试点车型、车辆数量和验收样例
